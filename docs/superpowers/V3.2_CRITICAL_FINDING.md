# v3.2 Critical Finding — MCP Timeout Constraint

**Date**: 2026-04-27  
**Severity**: HIGH (blocks 100+ page file extraction in single call)  
**Status**: RESOLVED (architectural solution identified)

---

## Problem Statement

**Untitled UI PRO extraction (113 pages, 12 batches) consistently fails with "MCP transport dropped mid-call" error.**

### Observed Behavior

| Attempt | Batches | Duration | Result |
|---|---|---|---|
| Full extraction (batches 1-12) | 12 | ~178s projected | ❌ MCP transport dropped |
| Chunked extraction (batches 1-3) | 3 | 44.6s | ✅ Success (2,271 components) |
| Resume attempt (batches 4-6) | 3 | Unknown | ❌ MCP transport dropped |

### Root Cause

**MCP `use_figma` calls have a hard timeout limit of ~45-60 seconds.**

- Single-batch average time: ~14.9 seconds
- Maximum batches per call: ~3 batches (before timeout)
- Files requiring 4+ batches: **CANNOT be extracted in a single MCP call**

This is NOT a Figma API timeout (which is handled by retry logic). This is an **MCP transport timeout** — the connection between Claude and the MCP server drops after ~45-60 seconds of continuous execution.

---

## Why This Wasn't Caught Earlier

**TC1 and TC2 validation (Carbon Design, 54 pages, 6 batches):**
- Validated 1-2 batches only (~28.6s for 2 batches)
- Did NOT attempt full 6-batch extraction in a single call
- Projected full extraction time (~85s) but did not execute

**False assumption**: We assumed the pagination logic would allow unlimited batches per call as long as individual batches stayed under timeout. **This assumption is incorrect.**

**The MCP timeout is on the TOTAL `use_figma` call duration, not per-batch.**

## Architectural Solution

### Current Implementation (v3.2.0 — BROKEN)

```
┌─────────────────────────────────────────────────┐
│  design-nexus skill (Claude environment)        │
│                                                 │
│  ┌───────────────────────────────────────────┐ │
│  │ Single use_figma call                     │ │
│  │                                           │ │
│  │  for (batch 1 to 12) {                    │ │
│  │    extractBatch()  // 14.9s avg          │ │
│  │  }                                        │ │
│  │                                           │ │
│  │  Total: ~178s  ❌ MCP TIMEOUT             │ │
│  └───────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

**Problem**: Entire extraction runs inside a single `use_figma` call, which times out after ~60s.

---

### Correct Implementation (v3.2.1 — PRODUCTION FIX)

```
┌─────────────────────────────────────────────────┐
│  design-nexus skill (Claude environment)        │
│                                                 │
│  for (chunkNum 1 to 4) {                        │
│    ┌─────────────────────────────────────────┐ │
│    │ use_figma call (chunk)                  │ │
│    │                                         │ │
│    │  extractBatches(start, end)             │ │
│    │    // 3 batches per call (~45s)         │ │
│    │                                         │ │
│    │  return partialSnapshot                 │ │
│    └─────────────────────────────────────────┘ │
│                                                 │
│    savePartialSnapshot(chunkNum)                │
│  }                                              │
│                                                 │
│  mergeAllPartials()                             │
│  cleanup()                                      │
└─────────────────────────────────────────────────┘
```

**Key change**: **design-nexus skill orchestrates MULTIPLE `use_figma` calls**, each handling a small chunk of batches (2-3 batches per call, staying under 60s limit).

## Implementation Changes Required

### 1. Refactor `extractWithPagination()` in SKILL.md

**Current** (broken):
```javascript
async function extractWithPagination(fileKey, pageCount, batchSize) {
  // Single use_figma call extracts ALL batches
  const result = await useFigma({
    fileKey,
    code: `
      for (let batch = 1; batch <= totalBatches; batch++) {
        extractBatch(batch);  // This loop is INSIDE use_figma
      }
    `
  });
}
```

**Correct** (production):
```javascript
async function extractWithPagination(fileKey, pageCount, batchSize) {
  const totalBatches = Math.ceil(pageCount / batchSize);
  const BATCHES_PER_CHUNK = 2;  // Conservative: 2 batches per MCP call (~30s)
  const chunks = Math.ceil(totalBatches / BATCHES_PER_CHUNK);
  
  const partialSnapshots = [];
  
  // Orchestrate multiple use_figma calls (one per chunk)
  for (let chunkNum = 1; chunkNum <= chunks; chunkNum++) {
    const startBatch = (chunkNum - 1) * BATCHES_PER_CHUNK + 1;
    const endBatch = Math.min(startBatch + BATCHES_PER_CHUNK - 1, totalBatches);
    
    console.log(`\n🔄 Extracting chunk ${chunkNum}/${chunks} (batches ${startBatch}-${endBatch})...`);
    
    // Each use_figma call handles 2 batches (~30s, well under MCP timeout)
    const chunkResult = await useFigma({
      fileKey,
      code: `
        // Extract batches ${startBatch} through ${endBatch}
        const components = {};
        for (let batch = ${startBatch}; batch <= ${endBatch}; batch++) {
          await extractBatch(batch, components);
        }
        return { components };
      `
    });
    
    // Save partial snapshot to disk after each chunk
    const partialPath = `.design-nexus/snapshots/.partial/${fileKey}-chunk-${chunkNum}.json`;
    await writeJSON(partialPath, chunkResult.components);
    partialSnapshots.push(partialPath);
    
    console.log(`  ✓ Chunk ${chunkNum} saved (${Object.keys(chunkResult.components).length} components)`);
  }
  
  // Merge all partial snapshots
  console.log(`\n🔗 Merging ${partialSnapshots.length} partial snapshots...`);
  const mergedSnapshot = await mergePartials(partialSnapshots);
  
  // Save final snapshot
  const finalPath = `.design-nexus/snapshots/${fileKey}.json`;
  await writeJSON(finalPath, mergedSnapshot);
  
  // Cleanup partials
  console.log(`🧹 Cleaning up partial snapshots...`);
  for (const partialPath of partialSnapshots) {
    await fs.unlink(partialPath);
  }
  
  return mergedSnapshot;
}
```

### 2. Key Differences

| Aspect | v3.2.0 (Broken) | v3.2.1 (Fixed) |
|---|---|---|
| **use_figma calls** | 1 (handles all batches) | Multiple (2-3 batches each) |
| **Loop location** | Inside use_figma code | In design-nexus skill |
| **Timeout risk** | HIGH (178s for 12 batches) | LOW (30s per chunk) |
| **Resume capability** | Not possible (single call) | Automatic (chunk-level) |
| **Partial snapshots** | In-memory only | Saved to disk per chunk |

### 3. Benefits of Chunked Approach

✅ **MCP timeout eliminated**: Each chunk stays under 60s limit  
✅ **Automatic resume**: If chunk N fails, restart from chunk N (N-1 chunks already saved)  
✅ **Memory efficient**: Each chunk releases memory before next chunk  
✅ **Progress transparency**: User sees chunk-level progress, not just batch-level  
✅ **Scalable**: Works for files of ANY size (1 page or 1,000 pages)

## Recommendation

**DO NOT SHIP v3.2.0 in current state.** The single-call pagination approach is fundamentally incompatible with MCP timeout constraints.

**Implement v3.2.1 (chunked approach) before shipping.** This is a **1-2 hour fix** that makes the feature production-viable.

**Alternative (NOT RECOMMENDED)**: Ship v3.2.0 with a documented limitation: "Pagination only works for files with <60 pages (6 batches or fewer)." This would make the feature useless for the exact systems it was designed to support (Untitled UI PRO, Flowbite Pro).

---

## Lessons Learned

1. **Always test at target scale.** Validating 2 batches and projecting to 12 batches is NOT the same as validating 12 batches.

2. **MCP timeouts are different from API timeouts.** The 60-second MCP transport timeout is a HARD limit on tool call duration, regardless of retry logic.

3. **"DEFERRED" can hide critical bugs.** TC4 (Resume Capability) was marked "DEFERRED" as an optimization, but it's actually **essential** for the feature to work at all on large files.

4. **Projection ≠ Validation.** "Projected duration: ~178s" looked acceptable in theory, but the MCP timeout limit at ~60s made it impossible in practice.
