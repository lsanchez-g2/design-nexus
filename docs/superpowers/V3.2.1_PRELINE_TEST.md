### Session Capacity Discovery

**Documented expectation** (from v3.2.1 design):
- MCP timeout: ~60s per call
- Session timeout: cumulative across calls
- Capacity: 2-3 chunks before timeout

**Actual result**:
- **5 chunks** extracted without timeout
- **No MCP timeout** warnings
- **Single session** completed entire extraction

**Hypothesis**: Read-only operations (Get API calls) have longer session capacity than write operations (Set API calls). The v3.2.1 documentation was based on write operation testing (Untitled UI PRO partial extraction with writes).

**Recommendation**: Update v3.2.1 docs to distinguish read vs write session capacity.

## Findings & Insights

### 1. Session Capacity (Read vs Write)

**Finding**: Read-only extractions have **longer session capacity** than documented.

**Evidence**:
- Documented: 2-3 chunks (based on write operation testing)
- Actual: 5+ chunks (no timeout on 97-page file)
- Operation type: Read-only (no writes to Figma)

**Hypothesis**: MCP session timeout is cumulative across write operations, but read operations have higher tolerance.

**Recommendation**: Test write-heavy workflows (creating components, setting properties) to measure write operation session capacity separately.

---

### 2. Non-shadcn/ui System Handling

**Finding**: design-nexus correctly identifies and documents non-shadcn/ui systems.

**Evidence**:
- Compliance score: 11% (accurate)
- Violations flagged: Prop naming (`Type` vs `variant`), value naming (`Solid` vs `primary`)
- Documentation: Violations noted in each component spec

**Validation**: Preline UI is a Tailwind-based system with custom conventions. design-nexus did NOT force shadcn/ui conventions — it documented the actual system faithfully while flagging differences.

**Recommendation**: Add "adaptation layer" guidance for non-shadcn/ui systems (how to map Preline UI conventions to shadcn/ui if needed).

---

### 3. Variant Matrix Completeness

**Finding**: Theoretical variant counts often exceed actual variant counts.

**Examples**:
- Button: 504 theoretical, 492 actual (12 missing)
- Avatar: 400 theoretical, 192 actual (208 missing)
- Alert: 48 theoretical, 30 actual (18 missing)

**Reason**: Intentional design choices — some variant combinations are semantically invalid or poor UX:
- White style × Light color = poor contrast
- Link style may not need all states
- Some Type × Color combinations don't make sense

**Assessment**: Expected behavior. Missing variants are **intentional exclusions**, not bugs.

---

### 4. Icon Collection Handling

**Finding**: Batch 9 (Icon Collection page) had 1,318 components (largest single batch).

**Challenge**: Icon collections are component-heavy but simple (no variants).

**Performance**: Extraction still completed in ~10s, no timeout.

**Recommendation**: Consider special handling for icon pages (skip if not needed for /specs/).

---

### 5. Component-Specific Token Scopes

**Finding**: Most tokens use `ALL_SCOPES` (non-optimal).

**Evidence**: From variable extraction, color variables have `scopes: ["ALL_SCOPES"]`

**Impact**: Every property picker shows all tokens (cluttered UX in Figma)

**Best practice**: Scopes should be specific:
- Background colors: `["FRAME_FILL", "SHAPE_FILL"]`
- Text colors: `["TEXT_FILL"]`
- Border colors: `["STROKE_COLOR"]`
- Spacing: `["GAP", "PADDING"]`

**Recommendation**: Document scope best practices in /specs/ generation (token bindings section).

## Recommendations

### For v3.2.1 Documentation

1. **Clarify session capacity**: Distinguish read-only vs write operation capacity
   - Read-only: 5+ chunks (validated on 97-page file)
   - Write-heavy: 2-3 chunks (documented from earlier testing)

2. **Add non-shadcn/ui guidance**: How to document systems with different conventions

3. **Include adaptation layer docs**: How to map non-shadcn/ui systems to shadcn/ui if needed

### For v3.3 (Future)

1. **Dynamic chunk sizing**: Adjust batch size based on page content density
   - Icon pages: Larger batches (low complexity)
   - Component pages: Smaller batches (high complexity)

2. **Scope validation**: Check token scopes during extraction, flag `ALL_SCOPES` usage

3. **Parallel extraction**: If MCP supports concurrent calls, extract chunks in parallel

4. **Progress visualization**: Real-time progress bar during extraction (already implemented in skill)

### For Users

1. **Small files (<50 pages)**: Use single-pass mode (no pagination)
2. **Medium files (50-100 pages)**: Single session, 3-5 chunks, auto-orchestration
3. **Large files (100-200 pages)**: Expect multi-session workflow (not tested yet)
