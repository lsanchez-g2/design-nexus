## Test-Driven Feature Recommendations

### 🎯 Priority 1 — Adaptive Batch Sizing

**Problem**: Fixed 10-page batch size causes inconsistent chunk duration (2-16s observed).

**Evidence from tests**:
- Flowbite Batch 4 (pages 30-39): 7.5s — high component density
- Flowbite Batch 8 (pages 70-79): 10.3s — complex marketing layouts
- Flowbite Batch 10 (pages 90-99): 15.9s — template pages (0 components but long duration)

**Root cause**: Component density varies dramatically between pages. 10-page batches work well for foundation pages (~2s) but become slow for component-dense pages (>10s).

**Solution**: Dynamically adjust batch size based on component density.

**Implementation**:
1. **Pre-scan phase**: Quick page scan (1-2s) to estimate component density per page
2. **Adaptive sizing**: 
   - Foundation pages: 20 pages per batch
   - Low-density pages (<5 components/page): 15 pages per batch
   - Medium-density pages (5-20 components/page): 10 pages per batch (current)
   - High-density pages (>20 components/page): 5 pages per batch
3. **Chunk target**: Maintain ~25s per chunk average

**Expected impact**:
- More consistent chunk durations (20-30s instead of 10-60s)
- Reduced worst-case extraction time by 20-30%
- Better MCP timeout safety margin

**Complexity**: Medium  
**Value**: HIGH — Improves user experience and extraction efficiency

---

### 📊 Priority 2 — Progress Indicators

**Problem**: Users have no visibility into extraction progress during long-running operations.

**Evidence from tests**:
- Flowbite Pro (208 pages): ~130s estimated total, no progress feedback
- Users don't know how many chunks remain, current batch status, or ETA

**Solution**: Real-time progress reporting during extraction.

**Implementation**:
1. **Batch-level progress**: "Batch 3/21 (pages 20-29): Extracting components..."
2. **Chunk-level progress**: "Chunk 2/11 complete (~30% done)"
3. **ETA calculation**: Based on average batch duration so far
4. **Component count**: "Extracted 47 component sets so far..."

**Output format**:
```
🔄 Extracting Flowbite Pro (208 pages)

Chunk 2/11 (~20% complete)
├─ Batch 3/21 (pages 20-29) ⏳ Extracting...
├─ Batch 4/21 (pages 30-39) ⏸️ Pending
└─ Component sets extracted: 47

Estimated time remaining: ~90s
```

**Expected impact**:
- User confidence (visible progress)
- Better time management (accurate ETAs)
- Early detection of issues (stuck batches)

**Complexity**: Low  
**Value**: MEDIUM — Improves user experience significantly

---

### ♻️ Priority 3 — Partial Extraction Resume

**Problem**: If extraction is interrupted (user cancels, network failure, crash), must restart from scratch.

**Evidence from tests**:
- Flowbite Pro (208 pages, 11 chunks): If interrupted at chunk 7, lose 6 chunks (~60s) of work
- No mechanism to resume from last completed chunk

**Solution**: Save partial snapshots after each chunk, allow resume from interruption point.

**Implementation**:
1. **Auto-save after each chunk**: Write chunk data to `.design-nexus/snapshots/.partial/[fileKey]-chunk-N.json`
2. **Detect interruption**: Check for partial snapshots on next invocation
3. **Offer resume**: "Found partial extraction (7/11 chunks complete). Resume? (yes/no)"
4. **Merge on completion**: Combine all chunks into final snapshot, delete partial files

**Workflow**:
```
Session 1: Chunks 1-7 complete → interrupted
Session 2: "Resume from chunk 7? (yes/no)" → yes → Chunks 8-11 → merge → complete
```

**Expected impact**:
- No wasted work on interruptions
- Enables pause/resume workflows
- Resilience to transient failures

**Complexity**: Medium  
**Value**: HIGH — Critical for very large files (200+ pages)

---

### 🧠 Priority 4 — Component Density Detection

**Problem**: Cannot predict extraction time accurately because density varies.

**Evidence from tests**:
- Preline UI: High density (2,301 variables, 4,165 components) → 1.33 pages/sec
- Flowbite Pro: Moderate density (161 variables, ~250 sets) → 1.59 pages/sec
- Batch duration range: 2-16s (8× variance)

**Solution**: Pre-analyze file to estimate extraction time and optimize batch sizes.

**Implementation**:
1. **Quick scan** (first 10 pages): Count components, variables, instances
2. **Density calculation**: 
   - Low: <5 components/page, <100 variables total
   - Medium: 5-20 components/page, 100-1000 variables
   - High: >20 components/page, >1000 variables
3. **Batch size adjustment**: Use adaptive sizing (Priority 1)
4. **ETA estimation**: `fileSize × avgSecondsPerPage` where `avgSecondsPerPage` based on density

**Output**:
```
Analyzing Flowbite Pro (208 pages)...
├─ Component density: Medium (7 components/page avg)
├─ Variable density: Low (161 variables)
├─ Estimated extraction time: ~130s
└─ Batch size: 10 pages (standard)
```

**Expected impact**:
- Accurate ETAs before extraction starts
- Better batch size decisions
- Informed user expectations

**Complexity**: Low-Medium  
**Value**: MEDIUM — Improves UX, enables Priority 1

## Additional Features (Lower Priority)

### 🔀 Priority 5 — Parallel Chunk Extraction

**Problem**: Chunks extract sequentially, even though they're independent.

**Potential**: If chunks could extract in parallel, 208-page file could extract in ~40s (3 chunks in parallel) instead of ~130s.

**Challenges**:
- Figma API rate limits (unknown)
- MCP tool call concurrency limits
- Complexity of parallel session management

**Status**: Research phase — needs spike to validate feasibility

**Value**: HIGH if feasible — 3-5× speedup potential  
**Complexity**: HIGH — parallel session orchestration, race conditions, error handling

---

### 📈 Priority 6 — Streaming Extraction

**Problem**: User sees no output until chunk completes (~30s latency).

**Solution**: Stream component data as each page completes within a chunk.

**Expected impact**: Perceived performance improvement (faster feedback)

**Value**: LOW — Nice-to-have, not critical  
**Complexity**: Medium

---

### 🎨 Priority 7 — Convention Auto-Fix

**Problem**: design-nexus detects convention violations (8-11% shadcn/ui compliance) but doesn't fix them.

**Solution**: Automatically convert Tailwind conventions to shadcn/ui during spec generation.

**Example**:
- Input (Figma): `Color=Primary`, `Size=base`
- Output (spec): `variant="primary"`, `size="default"`

**Challenges**:
- Semantic mapping ambiguity (`Alternative` → `secondary` or `outline`?)
- User may WANT Tailwind conventions (don't force shadcn/ui)
- Requires convention preference flag

**Value**: MEDIUM — Reduces manual spec editing  
**Complexity**: Medium — Requires mapping logic and user preferences
